Eu comecei a ouvir _lofi_ como algo que eu colocava de fundo na hora de estudar na universidade. Depois de me formar, também ouvia as mesmas músicas enquanto trabalhava ou precisa focar em algo. Foi aí que eu percebi que eu gostava de uma quantidade dessas músicas e que poderia fazer uma playlist com elas.

Nesse ponto, realmente fazer _só_ uma playlist seria mais que suficiente. Porém, como um engenheiro de software que brisa um pouco demais as vezes, pensei em construir uma **rádio inteira a partir de zero**. 

Este é um post sobre o processo de construção da rádio que aprofunda em vários conceitos de matemática e programação. Todo o código que você vê aqui é _open-source_, e portanto aberto para que qualquer pessoa possa construir sua própria versão se quiser.

# Client side

A parte do _client_ consome 2 APIs principais. 

* Audio _stream_ via requisição HTTP que retorna uma _stream de bytes_.
* Metadados sobre as músicas via _Server Sent Events (SSE)._ Envia eventos para o _client_ com informações sobre a música que está tocando no momento
* Metadata information about the track via Server Sent Events (SSE). It pings the client with the information about the current track being played.

Se vc colocou a rádio pra tocar ali em cima, vc já está vendo os metadados no tocador, ou do lado direito do site se está no _desktop_ ou uma faixa em baixo se está no celular.



A tag [audio](https://developer.mozilla.org/pt-BR/docs/Web/HTML/Reference/Elements/audio) é bem simples nesse momento, mas tem uma lógica em volta dele para lidar com algumas questões tipo reconexões com o servidor.

Por fim, também possui mecanismos para a tela não apagar em celulares e também usa a [MediaSession API](https://developer.mozilla.org/pt-BR/docs/Web/API/MediaSession) para transmitir as informações para os dispositivos, assim conseguimos ver o nome da música e artista no rádio do carro ou na tela do celular quando na tela de desbloqueio.

## Capas de álbuns e gradiente esfumaçado

Visualizar as capas de álbuns _enquanto_ a música toca é parte essencial da experiência. A arte de capa não esta ali por acaso geralmente e é criada das mesma maneira que as músicas são.

Usando isso como base, temos que **o elemento mais importante na página do rádio é essa capa.** É o maior elemento da página e em volta dela há um efeito gradiente de _blur_ das cores que são extraídas da imagem.

O efeito se chama `AmbientGlow` e fica no arquivo [src/lib/ambient-glow/index.ts](src/lib/ambient-glow/index.ts). Resumido, o efeito coloca a imagem em uma grade, extrai a cor predominante em cada segmento e desenha camadas de gradiente circulares atrás da imagem. Dê uma fuçada com os controles abaixo, cada _slider_ controla um dos argumentos de `GlowOptions` que o rádio passa na função de `mount()`.

### Como as cores são extraídas

Para cada "quadradinho" na imagem transformada em grade, o código chama `dominantColor()`, que faz um histograma dividindo os pixels em 3 cestas. A função anda por todos os pixels, 4 bytes por vez, filtrando todos que são muito transparentes, muito escuro ou muito brilhantes através uma verificação da _luminance_ da cor e por fim separando o que sobrou em nessas 3 cestas no formato RGB (uma para cada cor primária: vermelho, verde e azul).

```ts
const lum = 0.299 * r + 0.587 * g + 0.114 * b
if (skipBlack && lum < blackThreshold) continue
if (skipWhite && lum > whiteThreshold) continue
const key = `${Math.floor(r / binSize)},${Math.floor(g / binSize)},${Math.floor(b / binSize)}`
counts.set(key, (counts.get(key) || 0) + 1)
```

A cesta com maior número vence, e sua cor é reconstituída também em uma configuração tripla de `{ r, g, b }`. A variável `binSize` controla o quão "grande" são essas cestas: quanto maior são, mais cores e tons parecidos são unificados em uma cor dominante, cestas pequenas preservam as pequenas variações de cor ao custo de fragmentar a votação da cor dominante. Os filtros de preto e branco existem porque geralmente fundos muito claros ou muito escuros dominariam sempre na contagem e portanto removeriam as cores importantes do gradiente, que são justamente o que estamos procurando no algoritmo.

## Algo enviado a cada 26ms

Não exatamente a cada 26 milissegundos, mas esse número vem de como lemos os _frames_ de um arquivo MP3.

Voltando um pouco na história da internet: Os arquivos MP3 quando lançado revolucionou o jeito em que lidávamos com audio com uma eficiente compressão de tamanho enquanto mantém uma ótima qualidade de som.

![Um tocador MP3 antigo que revolucionou o conceito de musica portátil](/lofi/mp3-stick.webp)
(quem tinha um desses aparelhos sabe o que eu estou falando)

Dentro do formato MP3, há varios tipos de compressão de dados. A mais utilizada se chama _MPEG-1 Layer 3_, que é justamente a compressão que perde um pouco da qualidade da música mas que é tem um tamanho bem enxuto.

(need to also explain about the 44.1kHz sampling rate)

Para cada _frame_ de um áudio, ele armazena exatamente 1152 "pedaços" de áudio. POrtanto, se a música tem um _sampling rate_ de 44.1 kHz, cada frame precisa ter uma duração de 26ms (calc).

Cada frame de MPEG-1 Layer III carrega exatamente 1152 amostras de áudio. Divida pela taxa de amostragem e você tem quanto dura um único frame — a cadência que o servidor precisa manter.

Um frame sai a cada `26 ms` para _todos_ os ouvintes ao mesmo tempo. Se você chega atrasado, a música não recomeça — você entra exatamente onde ela está agora, igual a um rádio.

setTimeout works in a best-effort way when it comes to precise milliseconds. It can't guarantee something will way to be executed in exact 26ms. It can be 26, 24, 32ms, depending how soon the CPU can handle that process. We gotta remind ourselves that server aplications are CPU-bound and therefore share the Operational System resources like any other process running alongside it.

A cada `26 ms` o servidor precisa entregar um frame de MP3. Um loop ingênuo com `setTimeout` vai escorregando para fora da grade; o `PreciseTimer`, que se corrige pelo relógio, permanece travado nela.

buffer explanation

Uma stream em tempo real entrega o áudio rápido o suficiente para tocá-lo — _nunca_ mais rápido. A mesma rede instável alimenta os dois medidores; a única diferença é quanto há no buffer à frente. Com quase nada no buffer, cada engasgo o esvazia e você ouve uma falha. Alguns segundos de folga — o que o `burst-on-connect` instala no momento em que você conecta — absorvem os mesmos engasgos.

# Arquivos

Nessa versão atual da rádio, apenas o arquivo MP3 é suportado. O suporte de outras extensões e formatos requerem que a mesmo algoritmo que determina a duração de um _frame_ seja replicada para eles, para que a cadência de frames transmitidos na _stream_ seja constante.

A adição de arquivos é feita usando as APIs de Admin que conseguem gerenciar a playlist adicionando, editando metadados ou removendo músicas da playlist.

# Shelf para gerenciamento da playlist

As músicas que vc ouve aqui também estão disponíveis no Spotify e YouTube Music.
