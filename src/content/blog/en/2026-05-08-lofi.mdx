# Client side

The client side consumes 2 APIs:

* Audio stream via HTTP request that return a stream of bytes.
* Metadata information about the track via Server Sent Events (SSE). It pings the client with the information about the current track being played.

If you played the radio in the callout above, you are already seeing the metadata in the player either in the right side of the screen or on the bottom if you are on mobile.

The [audio](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/audio) tag itself is straightforward from this point, but we also have custom logic built on top of it to handle server reconnections.

Finally, we also have custom logic to prevent the screen to lock for mobile browsers and push metadata using [MediaSession API](https://developer.mozilla.org/en-US/docs/Web/API/MediaSession) to devices, so the information is visible in radio cars or lock screen in mobiles for example.
## Album cover and blur gradient

I believe artists put great thought into the album cover of their releases and the intention is to visualize it alongside the playing track. some of them are pretty dope as well.

Thinking about that, **the most important element on the radio page should be the album cover**. It is the biggest element there and around it there is a custom blur gradient effect from the colors that are extracted from the image.

The effect lives in [src/lib/ambient-glow/index.ts](src/lib/ambient-glow/index.ts). At a high level it samples the cover image on a small grid, picks one dominant color per cell, and paints a stack of soft radial gradients on layers sitting behind the image. ### How colors are extracted

For each cell in the sampling grid the code calls `dominantColor()`, which is just a histogram with three escape hatches. It walks the pixel buffer four bytes at a time, drops anything that's too transparent, too dark, or too bright via a simple luminance check, and then bins what's left into coarse RGB buckets:

```ts
const lum = 0.299 * r + 0.587 * g + 0.114 * b
if (skipBlack && lum < blackThreshold) continue
if (skipWhite && lum > whiteThreshold) continue
const key = `${Math.floor(r / binSize)},${Math.floor(g / binSize)},${Math.floor(b / binSize)}`
counts.set(key, (counts.get(key) || 0) + 1)
```

The bucket with the highest count wins, and its center is reconstructed back into an `{ r, g, b }` triplet. `binSize` controls how forgiving the buckets are — large bins merge close shades into one dominant color, small bins preserve fine variation at the cost of fragmenting the vote. The black/white skips exist because letterboxing and bright backgrounds otherwise dominate the count and drain the gradient of the colors that actually matter.

## something every 26ms

Not exactly every 26ms, but that number is coming from how we parse the MP3 frames.

So, a little back in history: The MP3 format itself is widely used to store and transport audio files. It is very efficient to compress an audio file size while maintaing the quality.

![Um tocador MP3 antigo que revolucionou o conceito de musica portátil](/lofi/mp3-stick.webp)
(the ones that had one of these know what I am talking about.)

Inside the MP3 format, there are several ways of compressing the data. the most used of them, MPEG-1 Layer 3, is a lossy compression that is very efficient in terms of compression ratio. This compression format states **each frame of an audio stores exactly 1152 samples**.

(need to also explain about the 44.1kHz sampling rate)

For each frame of an audio, it stores exactly 1152 audio samples. Therefore, if the track has 44.1kHz sampling rate, and MPEG-1 Layer 3 holds 1152 samples, each frame duration is ~26ms (need fuurther explanation about the frame duration, still confusing af).

Each MPEG-1 Layer III frame carries exactly 1152 audio samples. Divide by the sample rate and you get how long a single frame lasts — the cadence the server has to hold.

setTimeout works in a best-effort way when it comes to precise milliseconds. It can't guarantee something will way to be executed in exact 26ms. It can be 26, 24, 32ms, depending how soon the CPU can handle that process. We gotta remind ourselves that server aplications are CPU-bound and therefore share the Operational System resources like any other process running alongside it.

Every `26 ms` the server must hand out one MP3 frame. A naive `setTimeout` loop drifts off the grid; the clock-correcting `PreciseTimer` stays locked to it.

## A head start so it doesn't stutter

{/* DRAFT — burst-on-connect / buffer cushion. flesh this out. */}

draft notes: pacing one frame every 26ms is exactly real-time, never faster. so a listener who just connected has an empty buffer and joins right at the live edge — the same "everyone on the same frame" idea from the fan-out section. problem: with zero cushion, any network jitter drains the buffer to zero and the audio stutters (the browser fires a `waiting` event, you hear a "chop").

draft notes: the fix is burst-on-connect — the server keeps a small ring buffer of the most recently broadcast frames and replays it the instant a client connects, so the decoder starts with a few seconds of runway instead of nothing. same trick Icecast/Shoutcast use.

A real-time stream hands out audio just fast enough to play it — _never_ faster. The same jittery network feeds both gauges; the only difference is how much is buffered ahead. With almost nothing buffered, every hiccup drains it to empty and you hear a gap. A few seconds of cushion — what `burst-on-connect` installs the moment you connect — absorbs the same hiccups.

# Files

In this current version of the radio, it only supports MP3 files. In order to support more extensions, the same algorithm to determine the frame duration would need to be replicated to them so the frames constant cadence is maintainted in the stream level.

The addition of files is done using the Admin APIs that can manage the playlist by adding, editing metadata or removing songs in the playlist.
